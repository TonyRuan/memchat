# 开发交付流程偏好

## 类型

preference

## 状态

active

## 可信度

90

## 摘要

用户希望 MemoryChat 项目中影响应用行为或可测试产物的有效变更都走完整交付闭环：同步版本号、写开发日志、用 Git 保存；关键交付最好构建 Debug APK，方便直接安装测试。纯审计报告、规划文档或只读分析不应单独 bump APK 版本。

## 适用边界

- 适用于当前项目的代码、配置、产品行为和可测试产物变更。
- 纯审计报告、规划文档和只读分析可以 Git 保存并写清上下文，但不应单独更新应用版本号。
- 纯探索、只读分析、临时排查不需要更新版本号或构建 APK。
- 涉及本机私密配置时，不应把密钥或本地配置提交进 Git。

## 当前执行规则

- 影响应用行为或可测试产物的有效项目变更交付前，同步更新 `app/build.gradle` 中的 `versionCode` 和 `versionName`。
- 每次有效变更都在 `DEVLOG.md` 写清楚版本、变更内容、验证结果和 APK 输出情况。
- 每次有效变更完成验证后，用 Git 进行一次范围清晰的提交。
- 关键功能、体验、配置或验收相关交付，构建 Debug APK 并在交付说明中给出路径。

## 触发模式

- 用户要求实现、修复、配置或文档化会影响项目后续状态的内容。
- 变更会改变应用行为、配置默认值、构建产物或调试流程。
- 用户要求交付可测试版本或需要在设备上验证。

## 可复用规则

- 影响应用行为或可测试产物时，把版本号、开发日志、验证结果和 Git 提交作为同一条闭环处理。
- 对关键交付默认产出可安装 Debug APK，避免只给代码差异而无法快速测试。
- Git 提交保持范围清晰，只包含本次相关文件；敏感本机配置继续留在忽略文件中。

## 跨项目复用测试

去掉 MemoryChat、Android、Gradle 等项目细节后，这条偏好仍成立：对于用户反复调试的本地产品项目，有效变更应能追溯版本、日志、提交和可测试产物。
