# PRD 对齐的子代理测试偏好

## 类型

preference

## 状态

active

## 可信度

90

## 摘要

用户希望 MemoryChat 后续测试工作尽量用子代理并行完成，以提升速度并避免主上下文污染；已有测试可能不全面、不严谨，后续应结合 PRD 和项目实际行为进行测试重构。

## 适用边界

- 适用于测试审查、测试重构、验收覆盖率分析、CI/构建验证和 E2E 复核。
- 简单单文件验证或快速复现可直接在主线程执行，但关键测试策略和跨模块覆盖应优先拆给子代理。
- 子代理结果不能直接视为最终结论，主线程仍需整合、复核并跑完整验证。

## 当前执行规则

- 测试任务能安全拆分时，优先按 PRD 章节、测试层级或模块边界派发子代理。
- 测试重构不能只沿用旧用例名称，要从 PRD P0、最小通过标准和真实代码行为重新建立覆盖矩阵。
- 子代理负责只读审查或互不冲突的测试切片，主线程负责整合、去重、验证和最终提交。
- 旧测试若只验证实现细节、缺少断言、与 PRD 脱节或不能稳定复现，应重写或降级为辅助测试。

## 触发模式

- 用户要求补测试、重构测试、提高测试严谨性或按 PRD 验收。
- 变更影响聊天、记忆、人格、上下文压缩、导入导出、安全或 Debug 展示。
- 需要同时检查多个测试域、多个模块或多个验收场景。

## 可复用规则

- 先建立 PRD 覆盖矩阵，再决定单元测试、仪器测试和人工 E2E 的边界。
- 并行子代理适合做独立审查、覆盖映射和互不冲突的测试补丁。
- 最终必须由主线程跑项目级验证，不能只采信子代理报告。

## 跨项目复用测试

去掉 MemoryChat、PRD 文件名和 Android 细节后，这条偏好仍成立：当用户认为旧测试不全面或不严谨时，应优先用并行审查降低盲区，并以真实需求和系统行为重建测试覆盖。
